# 日本語の技術文書向け文章規範

日本語で技術的な文章を書く・推敲するときは、以下の規範に従う。これは、媒体を問わず、内容のない型や過剰な演出を避け、人間が状況と読み手に合わせて書いたような文章にするための共通規範である。

各規範は、対象となる内容があるときだけ適用する。規範を満たすために、短いIssueへ見出しを足したり、存在しない反論や例を作ったりしない。

技術書の章や公開記事など、複数の節にわたって論旨や事例を展開する長文原稿では、保存場所を問わず、共通規範に加えて[原稿向け文章規範](references/manuscript.md)を読む。
Issue、PR、セットアップ手順、runbook、APIリファレンスなど、作業や参照を主目的とする文書には、原稿向け文章規範を適用しない。

## 執筆と推敲の順序

1. 文書の用途、読み手、既存の規約を確認する。
2. ユーザーから渡された情報、差分、テスト結果、ログ、既存文書から事実を集め、確認済みの事実、推定、未確認事項を分ける。
3. 読み手が判断または行動するために必要な情報を選び、その情報に合う最小限の構造で書く。
4. 内容の正確さと意味を確認してから、語調、冗長性、表記を整える。自然に見せるために事実や不確実性を変えない。

## 用途に応じた構造

リポジトリの`CONTRIBUTING.md`、テンプレート、既存文書、formatter、linterに規約がある場合は、それを優先する。

- Issueでは、読み手が1つのまとまりとして追跡できる問題や要望を扱う。複数の項目をまとめる追跡用Issueなど、ユーザーが定めたスコープを文章上の都合で変えない。バグ報告では、判明している現象、再現手順、期待する結果、実際の結果、環境、ログを、再現や切り分けに使える範囲で分ける。短いIssueに固定の見出しを強制しない。
- PRでは、目的、変更内容、変更理由、確認結果、関連Issueを、レビューに必要な範囲で示す。必要なら、特に見てほしい箇所、既知のリスク、レビュー順も添える。不可分な変更やユーザーが定めたスコープを保ちつつ、目的と変更の関係が追えるようにする。
- リポジトリ内の文書では、文書の目的に応じて説明、手順、リファレンスを構成する。順序のある操作は番号付きリスト、順序のない列挙は箇条書き、複数の属性を比較する情報は表を使う。
- 短さのために必要な文脈を落とさない。読み手が判断、再現、レビュー、操作に必要な情報を残したうえで、関係のない詳細を省く。

## Markdownの構造

- 段落は空行で分ける。IssueやPRの通常の段落では、文ごとの単一改行を使わない。
- `.md`ファイルの段落内改行と行長は、既存文書、formatter、linterの規約に従う。規約がなければ、一文一行や固定幅の折り返しを機械的に強制しない。
- 識別子、パス、短いコマンドはインラインコードで示す。複数行のコード、差分、ログ、設定、または単独でコピーする内容はコードブロックで示す。

## 表記上のハウススタイル

明示された表記規約がない場合は、以下をこのskillのハウススタイルとして使う。既存の文章を推敲するときは、意味のある表記や書き手の癖を、ハウススタイルだけを理由に均さない。

- 定義や分類の列挙は箇条書きで示してよい。本文で意図的に定義する用語は、走査しやすくなる場合に初出を太字にしてよい。導入後は通常の表記に戻す。語そのものを話題にするメタな言及、引用、通称には「」を使う。
- ダッシュ（emダッシュ`—`、horizontal bar`―`、いわゆる2倍ダッシュ「——」）を、内容のない区切りや強調のために日本語の地の文や見出しへ足さない。書き換える場合は、同格や補足の挿入を括弧にし、言い換えや敷衍を二文に分けるか読点でつなぐ。原文の語り口、公式表記、範囲を示すenダッシュ`–`、英語の複合語（`Curry–Howard`など）、コード、書誌情報は保つ。
- 中黒（・）を日本語の並列へ機械的に使わない。助詞や読点のほうが関係を明確にできる場合は書き換える。簡潔な並列として読みやすい場合と、固有名詞、公式名称、定着した複合語、UI表記、引用、書誌情報では使ってよい。
- 見出しに、区切り線（罫線`─`（U+2500）やダッシュ類）で`種別──主題`や`主題──概念`のように2要素を詰め込まない。見出しは単一の自然な句にする（要素を1つに絞るか、助詞や読点でつなぐ）。
- 用語と定義を箇条書きで並べる場合は、区切り線ではなく全角コロンで「**用語**：説明」と書いてよい。
- 人名表記は、その分野、媒体、既存文書の慣用に合わせる。検索や人物の同定に原綴りが必要なら、欧文人名の初出で定着した日本語表記と併記する。

## 段落と情報構造

- 段落には中心となる話題を持たせる。背景、現象、原因、対応、検証などが混ざって追いにくい場合は、段落やリストを分ける。導入から帰結までを同じ段落に置くほうが自然なら、機械的に分断しない。
- 複数文からなる説明段落では、読み手が主題を見失う場合に、冒頭付近で主題を明らかにする。すべての段落を同じ形の主題文から始めない。
- 文脈や語順だけでは前の情報との関係が伝わらない場合は、見出し、ラベル、箇条書き、接続表現のいずれかを使う。
- 文脈のない辞書型の定義を連ねない。用語は必要になる位置で定義し、役割や既存概念との差が判断に必要なら、その説明も添える。語の境界を先に示す必要がある場合や、用語集、リファレンスでは、簡潔な定義から始めてよい。
- 誤った解釈や主張を否定する必要がある場合は、否定する内容と根拠を具体的に書く。「何もかも解決するわけではない」のような漠然とした否定や、理由のない「AではなくB」で済ませない。必要なら、「もしAなら、別の結果になるはずだ」のような反実仮想で違いを示す。
- 譲歩で認めるのは、事実または成立する条件に限る。あとで訂正する因果関係をいったん書き手の判断として断定しない。よくある解釈を示す必要があるなら、その解釈の主体や根拠を明らかにする。
- 「ただし」「とはいえ」などの譲歩や限定には、その後の判断に必要な役割を持たせる。逆接を置くだけで話を宙吊りにしない。

## 正確さと根拠

書き上げたら、主張と根拠の対応、不確実性、成立条件、反例となる条件を点検する。

- 断定の強さを、文の役割と根拠の強さに合わせる。
  確認済みの事実、再現条件、検証結果は具体的に述べる。
  未確認事項、ログや観測結果からの推定、将来の見通し、仮定に基づく説明では、その状態を明示する。
  `かもしれない`、`だろう`、`ようだ`、`らしい`は、根拠なく主張を弱めている場合だけ削る。
  `必須`、`保証する`、`解決する`は、成立条件と根拠を示せる場合だけ使う。
  文章を自然に見せようとして、禁止、必須、推奨の強さを変えない。
  悪い例：追加の根拠がないまま、`この変更が原因かもしれない`を`この変更が原因だ`に変える。
  良い例：「ログだけでは、この変更が原因である可能性を否定できない」のように、根拠に応じた不確実性を残す。
- 異なるものを「同じ」とまとめない。区別すべき対象（別々の決定、別々の原因、種類の違う問題）を一括りの言葉でくくらない。悪い例：相互依存する3つの未決事項を「同じ決定を別々に下していた」と書く。良い例：「どれも別々の決定であり、しかも互いに依存している」と腑分けする。
- 複数の要因がある事象を、単一の原因に還元しない。それぞれを切り分け、どの根拠、修正、検証がどの要因に対応するのかを示す。悪い例：設定の読み込み失敗と権限不足が混在する障害を、丸ごと「キャッシュの問題」と説明する。
- 文章全体で、同じ概念の扱いを一致させる。呼称の違いだけで不一致と判断せず、主体、手続き、責任など、説明している層を区別する。「担当者が提案し、チームで合意する」は矛盾しない。
- 複数の概念を1つの上位語にまとめるときは、共通点や帰着関係を示してから命名する。列挙の直後に、説明なしで新しい総称へ飛ばない。
- 読み手に自明でない、争点になる、または判断を左右する因果を主張するときは、必要な粒度で根拠または機構（なぜそうなるのか）を示す。根拠が直前やリンク先にある場合は、短い参照でよい。複雑な機構は、一文に押し込まず複数の文、箇条書き、図に分ける。悪い例：「手順で分けると変更が全体に波及する」。良い例：「各工程がデータを受け渡すための表現を共有してしまい、その表現を変えると全体に波及する」。
- 主張は、挙げた例が実際にその全体を支えているかを確認する。例が主張の一部しか支えないなら、主張の範囲を例に合わせて狭める。
- 読者に共有されていない語、一般的な用法と異なる語、対象範囲が曖昧な語は、初出付近で意味または対象範囲を示す。既存文書で定義済みなら、その文書へのリンクで代えてよい。

## 読み手の負荷の管理

読者の記憶と注意は有限の資源として扱う。

- 抽象的な言い回しの指す内容が文脈から一意に決まらないときは、具体的な名詞、同格の括弧、ラベル、箇条書き、リンクなどでその場で特定し、できるだけ読者に前を読み返させない。
- 複数の例や場面の関係が自明でないときは、前の例との違いと、別の例が必要な理由を示す。
- 導入では、これから扱う内容に関係しない過剰な詳細を詰め込まない。
- ファイル名、関数名、時刻、数値などの具体は、判断、再現、レビュー、後の参照に必要なものだけを残す。必要な証拠まで一般語へ置き換えず、役割のない装飾的な精度だけを省く。

## 視点と語り

- 対象を指す語は具体的に選ぶ。「AI」「ツール」のような広い語でぼかさない。
- 行為者が判断や結果の理解に関わる場合は、誰が何をしたかを明らかにする。「特定され、判明した」のような受動態の結果だけを連ねない。
- 例に必要のない架空の人物設定を足さない。「入社2年目のエンジニア」のような属性は、その属性が結果や判断に関係する場合だけ示す。
- 文章内で用語や分類名（破壊的変更、互換性、ロールバックなど）を定めたら、以後は同じ語で通す。「問題」「対応」「仕組み」のような曖昧な類語へ後退しない。用語を定める前の導入では、一般語を使ってよい。
- 定義済みの術語を使うこと自体は、LLMっぽい抽象表現とは見なさない。初出で意味を示し、以後は必要な術語を一貫して使う。
- 術語や訳語は、対象分野の公式文書、既存の用語集、説明している層に合わせる。意味の近い漢語を一般語の感覚だけで充てず、同じ英語に複数の訳語がある場合は、対象の違いを確認する。
- 術語の響きを持つ語を、術語でない場面に流用しない（システムから人間までの連なりを「経路」と呼ぶ、など）。「届くまでの流れ」「あいだに何があるか」のように普通の言い方で書く。
- 推敲では、元の一人称、敬体・常体、意図的な反復や問いかけなど、媒体に合った語り口を理由なく均さない。既存の文体を変える必要がある場合は、ユーザーの指定または媒体の規約を根拠にする。
- 文長、文末、段落長を機械的に揃えない。反復によって単調さや関係の誤読が生じている場合だけ、説明のまとまりに応じて文をつなぐか切る。変化を作るためだけに文末を言い換えたり、短文、体言止め、接続語を挿入したりしない。

## 誇張と演出の抑制

事実と根拠を中心に書く。強調や比喩は、内容を明確にする場合だけ使う。

- 本文中の太字強調を多用しない。警告、否定、帰結、定義語など、誤読を防ぐ場合や走査しやすくする場合に使う。それ以外は見出し、リスト、文の順序で際立たせる。
- 変更、発見、影響を過剰に劇的にしない。重要性は感嘆符ではなく、具体的な事実や影響範囲で示す。
- 影響を列挙して、失敗や危険を必要以上に煽らない。
- 慣用表現をひねった言い回し（「知識を体に入れる」など）や、指す内容が一意に決まらない比喩（「報告の外側に世界が広がっている」など）を使わない。平易な動詞でそのまま言う（「身につく」「気付く機会が減る」）。

## 空虚な定型表現の点検

LLMが大量生成する、中身のない型に誘惑されない。以下は語句そのものの禁止一覧ではない。削除しても主張、関係、範囲が変わらない使い方だけを削る。論理的な役割や具体的な情報を担う表現は残す。

次の表現は、論点や関係を具体化しない空虚な用法になっていないか点検する。

- 予告と総括：「重要なのは〜である」「このIssueでは〜を扱う」「このPRでは〜を行う」「ここでは〜について見ていく」（内容を加えない予告だけの場合）、「まとめると」「要するに」（直前の言い換えだけの場合）、「〜に他ならない」
- 正面から系：「正面から扱う」「正面から回収する」「正面から見る／書く／立てる」。中身の代わりに姿勢だけを宣言する
- 空虚な形容：「不可欠」「核心的」「鍵となる」「根本的な」（主張の中身を説明せず強調だけする）、「多角的」「包括的」「総合的」（何をどう見たかを書かない）
- 空虚な動詞：「掘り下げる」「深掘りする」「言語化する」（何をどう書いたかを示さず終わる）、「触れる」「言及する」（一段落で済ませるだけ）
- 接続の型：「〜において」「〜という側面から」「〜の観点から」（新情報がない場合）、「さらに」「また」「加えて」の連打
- 空虚な称賛：「非常に」「極めて」「大いに」（程度や比較対象を示さない強調）

悪い例：「このPRでは、この問題に正面から対応する」「このIssueを多角的に分析すると、重要なのは設定である」「ここでは根本的な改善について見ていく」。
良い例：「このPRは、WSLで重複していたNix設定を1つに統合する」「このIssueでは、`nix run .#switch`が権限不足で失敗する条件を整理する」「この文書は、GPG鍵がない環境でsecretsをスキップする手順を説明する」。

## 冗長の排除

無駄な文章をなるべく残さない。

- 同じ役割、同じ粒度の箇所で、同じ主張を言い換えて繰り返さない。分けることで理解しやすくならない事実や説明は、一文または1つのリストにまとめる。タイトルと詳細、概要と検証結果、冒頭の要点と末尾の帰結、独立して読まれる節の文脈回復など、走査、記憶、論証で役割の異なる再掲は許容する。
- 読み手が容易に補える中間段階は省いてよいが、判断に必要な前提、因果の橋渡し、操作手順は省かない。
- 同じ関係を繰り返し説明する場合は、構造に名前を付けて短く参照してよい。ただし、名前を付けるだけで意味や根拠の説明を省かない。
- `疑問に思ったかもしれない。そのとおりである`のように、想像上の読み手の反応を決めつけて問答を演じない。「ここまでの話には自然な続きがある」のようなメタな枠も、内容を増やさないなら省く。問いや対話形式が論点または場面に役割を持つ場合は使ってよい。
- 「この文書もそれを否定しない」のように、内容を増やさない書き手の立場の弁明を置かない。認めるべき条件や事実があるなら、それを直接書く。
- 接続や評価のためだけの文（「それ自体はよいことである」など）を置かない。
- 対比、因果、順序を明らかにする接続表現は、単に短くするために削らない。関係を示さない接続表現だけを省く。
- 項目数、段落の長さ、節の構成を機械的に揃えない。根拠なく三項にそろえたり、すべての節を予告、本文、要約の同じ型に押し込んだりしない。
- 文脈を示さずに、まだ導入していない概念や文書名を持ち出さない。

## 見出しの付け方

見出しは内容を特定できる具体的なものにする。その節で答える問い、扱う対象、または結論を示す。疑問形、名詞句、断定形のうち、文書の目的と本文のトーンに合う形を選ぶ。

- 「例に戻す」「確認する」のように書き手の操作だけを述べず、その節が扱う対象または答える問いを示す。

## 読者への誠実さ

- 確認していない内容を、確認済みの事実として滑らかに書かない。
- 例が作為的に見えうる場合は、その限界を隠さない。実例、観測結果、成立条件など、読み手が照合できる材料を示す。現実に当てはめられる範囲を判断できるようにする。
